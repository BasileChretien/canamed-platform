Offline editor for content packs (CASE + SCORING + PENALTIES + DECISIONS)

← Back to the platform  ·  Tool UI in English; the scenarios it produces have { en, fr, ja } fields.

How to use this tool
  1. Fill in the form below. Every text field that the platform shows to learners is split into English / French / Japanese — English is the canonical fallback when fr/ja are empty, so partial translations are safe.
  2. Use the Add row buttons to grow the history / exam / labs / prompts / scoring / penalties / decisions lists. Use the × on each row to remove.
  3. The Live JSON preview updates as you type. Click Validate to check ids are unique and the synthesis prerequisites resolve.
  4. Click Copy JSON snippet to copy a window.CANAMED_SCENARIOS["your-id"] = { … } block, ready to paste into case-content.js. Or Download .json for a standalone file. Load JSON round-trips an existing scenario back into the form for editing.
1. Scenario metadata

A branched scenario is a clean, one-decision-at-a-time fork: the team votes, locks in, sees what happens, then the case moves on. English-only — the in-app reader supplies French / Japanese at read-time.

Lowercase, hyphenated. Used as the database key; cannot collide with existing scenarios in case-content.js.

Naming a module is what switches it on. Fill in both to run the full Module A + B session, or name just one for a single-module session — the session then skips the other module's stage entirely. At least one is required.

Branch tree

Each node is one decision shown on its own. For every choice, write what happens next (the consequence) and where the case goes — pick another node under Then →, or leave it as ends the case. Add the nodes first, then wire the Then → targets. The first node nothing points to is the start. English only.

2. CASE.history — questions the team can ask the patient

Each row is a clickable button (q) revealing the patient’s answer (a). Order matters: index 0 is shown first. Indices are referenced by synthPrereqs and PENALTIES.item below as history:N.

3. CASE.exam — physical examination items

Same shape as history. Referenced as exam:N.

4. CASE.labs — investigations & clinical synthesis

Tick the key box on exactly one row — that is the gated synthesis item. Its index becomes synthId = "labs:N" below. Referenced as labs:N.

5. CASE.prompts — discussion questions revealed after synthesis
6. Synthesis gate

Auto-populated from the lab row whose key checkbox is ticked. You can override manually if needed.

The items that must be revealed before the synthesis unlocks. Each must resolve to an existing history:N, exam:N or labs:N.

7. SCORING.moduleA — concept families for stage 1

For each family pick EITHER any keyword stems (English only; comma-separated) OR tick cohorts for the “named two partner universities” family. If both are filled, cohorts wins.

8. SCORING.moduleA_questions — LLM chat consultation scoring

Points earned for good history-taking questions typed into the Module A patient chat. Same shape as moduleA families — any stems match the learner’s typed question. Leave empty if the scenario has no chat scoring.

9. SCORING.moduleA_question_penalties — chat deductions

Deductions for harmful or opioid-seeking prompts typed into the Module A patient chat (e.g. demanding a prescription on demand). Optional.

10. SCORING.moduleB — concept families for stage 2
11. PENALTIES — deductions for revealing the wrong item

Each penalty fires when the team reveals the item referenced. item takes the form group:index, e.g. labs:4.

12. DECISIONS — team votes

Each decision belongs to module A or B, has a prompt, and 2+ options. Tick the correct option(s). penalty: 0 = wrong vote costs nothing (useful for low-stakes roleplays).

13. preTest — knowledge check before Module A (optional)

Multiple-choice questions shown after Welcome, before Module A. Optional — leave empty to hide the pre-test panel. Each question has a stem, 2+ options (tick the correct one[s]), and an explanation shown after answering.

14. postTest — knowledge check during Wrap-up (optional)

Same shape as the pre-test; shown during Wrap-up to measure knowledge gain. Optional.

15. characters — LLM chat roles (optional)

The Module A patient chat is driven by a character here. Add one with role: patient and write its persona (the system prompt: who they are, why they’re here, how they behave). Leave empty to fall back to a neutral patient synthesised from the case history. The server prepends an un-overridable safety guard.
